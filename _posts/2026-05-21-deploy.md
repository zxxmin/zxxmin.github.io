# 환경 구분
| 환경      | 관련 파일                                                                  |
|---------|------------------------------------------------------------------------|
| **local** | `Dockerfile.local`,<br/>`docker-compose.local.yml`,<br/>`.env`         |
| **dev** | `Dockerfile.dev`,<br/>`docker-compose.dev.yml`,<br/>`.env.dev`,<br/>`build.sh` |

---

# 전체 배포 흐름 요약
```
[1단계 - 개발자 로컬]
./build.sh 실행
  ↓ .env.dev에서 환경변수 로드
  ↓ Dockerfile.dev로 Docker 이미지 빌드
  ↓ AWS ECR에 이미지 push (:latest + :{commit-id} 태그)
  
[2단계 - AWS ECR]
이미지 저장소 역할 (Docker Hub의 AWS 버전)

[3단계 - EC2 실서버]
ECR에서 이미지 pull
  ↓ docker compose -f docker-compose.dev.yml up -d
  ↓ 컨테이너 실행 (빌드 없이 이미지 그대로 실행)
  ↓ nginx:80 서빙 + 백엔드 컨테이너 리버스 프록시

```
> **핵심 개념**: 빌드는 로컬에서 한 번만 일어나고, EC2는 완성된 이미지를 실행

# 파일별 역할
## 1. `.env.dev` — 빌드 시점 환경변수 정의
```yaml
VITE_API_URL=/api
VITE_API_DS_URL=/ds-api
VITE_API_DE_URL=/de-api
```
- `build.sh`가 이 파일을 읽어 Docker 빌드 인수(`--build-arg`)로 전달
- 값이 `/api` 같은 경로 prefix인 이유:<br/>
  실제 백엔드 컨테이너 주소는 `nginx.conf`에 하드코딩되어 있고, 이 값들은 nginx의 `location` 경로와 매핑되는 **URL prefix**
- **주의**: `VITE_` 접두사가 붙은 환경변수는 Vite가 **빌드 시점**에 번들(JS 파일)에 직접 삽입. 컨테이너 실행 후 환경변수를 바꿔도 반영되지 않으므로, 반드시 빌드 인수로 전달.

---

## 2. `build.sh` — ECR push 자동화 스크립트
로컬에서 실행하는 **빌드 + ECR push** 스크립트.

```
[사전 검증]
git 레포 확인 / Dockerfile 존재 여부 / AWS CLI / Docker 설치 여부
    ↓
[1단계] ECR 로그인
aws ecr get-login-password | docker login
    ↓
[2단계] Docker 이미지 빌드
.env.dev 환경변수 로드 → docker build --build-arg로 전달
태그 2개 생성: :latest + :{commit-short-id}
    ↓
[3단계] ECR 태그 부착
로컬 이미지명 → ECR 레포지토리 이미지명으로 태그
    ↓
[4단계] ECR Push
:latest, :{commit-id} 두 태그 모두 push
```

### commit-id 태그를 함께 붙이는 이유:
`:latest`만 사용하면 어느 커밋 기준으로 배포됐는지 추적 불가. `:{commit-id}` 태그로 특정 버전으로 즉시 롤백 가능.

## 3. `Dockerfile.dev` — 실서버 배포용 멀티스테이지 빌드
```dockerfile
# 1단계: 빌드 (node:24 이미지)
FROM node:24 AS build
ARG VITE_API_URL          # build.sh의 --build-arg로 받은 값
ENV VITE_API_URL=$VITE_API_URL  # ARG → ENV로 재할당 (Vite가 ENV를 읽음)
RUN yarn build:docker     # dist/ 생성

# 2단계: 서빙 (nginx:alpine 이미지)
FROM nginx:alpine
COPY --from=build /app/dist /usr/share/nginx/html   # 빌드 결과물만 복사
COPY nginx.conf /etc/nginx/conf.d/default.conf       # nginx 설정 복사
EXPOSE 80
```

### 멀티스테이지 빌드의 장점:
- 1단계(빌드)에서 사용한 Node.js, 소스코드, node_modules가 최종 이미지에 포함되지 않음
- nginx:alpine 기반으로 이미지 크기 최소화 (수백 MB → 수십 MB)

###  `build:docker`와 `build`의 차이:
| 스크립트          | 명령 | 용도 |
|---------------|---|----|
| `build`       | `tsc -b && vite build` | 로컬: 타입 검사 포함 |
| `build:docker` | `vite build`  | Docker: 타입 검사 없이 빌드만 |

- Docker 빌드 시 tsc -b를 제외한 이유: 타입 오류로 CI 배포가 블로킹되는 것을 방지.

## 4. `nginx.conf` — SPA 라우팅 + 리버스 프록시 설정
```
# SPA 라우팅 지원: 새로고침 시 404 방지
location / {
    try_files $uri $uri/ /index.html;
}

# /api/* 요청 → backend 컨테이너로 프록시
location /api/ {
    proxy_pass http://backend:8080/;
}

# /ds-api/* 요청 → ML 컨테이너로 프록시
location /ds-api/ {
    proxy_pass http://crsn-platform-ml:8100/;
}

# /de-api/* 요청 → DE 컨테이너로 프록시
location /de-api/ {
    proxy_pass http://crsn-platform-de:8000/;
}
```

### 리버스 프록시 동작 원리:
```
브라우저 → /api/users 요청
  ↓ nginx가 /api/ prefix 감지
  ↓ http://backend:8080/users 로 전달 (컨테이너 이름으로 통신)
  ↓ 응답을 브라우저에 반환

```

- `backend`, `crsn-platform-ml`, `crsn-platform-de`는 Docker 컨테이너 이름
- 같은 `shared-network` 안에 있어야 컨테이너 이름으로 통신 가능 (DNS 역할)
- CORS 문제 없이 백엔드 통신 가능 (브라우저 입장에서 같은 도메인)

## 5. `docker-compose.dev.yml` — 실서버에서 컨테이너 실행
EC2에서 컨테이너를 띄울 때 사용하는 파일. **빌드는 하지 않고 ECR 이미지를 그대로 실행.**

```yaml
services:
  front:
    build:
      context: .
      dockerfile: Dockerfile.dev
      args:
        VITE_API_URL: /api        # 빌드 인수 (ECR 이미지 사용 시 불필요)
    container_name: Crsn_FE-dev
    networks:
      - shared-network            # 백엔드 컨테이너들과 같은 네트워크
    ports:
      - "80:80"                   # 호스트 80 → 컨테이너 80 (nginx)
networks:
  shared-network:
    external: true                # 외부에 미리 생성된 네트워크 사용

```

### shared-network를 external로 쓰는 이유:
- `backend`, `crsn-platform-ml` 등 다른 백엔드 컨테이너들도 같은 네트워크에 연결되어 있어야 nginx가 컨테이너 이름으로 접근 가능
- `external: true`는 "이 compose가 네트워크를 직접 만들지 말고, 이미 만들어진 네트워크를 사용하라"는 의미

## 6. `Dockerfile.local` + `docker-compose.local.yml` — 로컬용
```yaml
# docker-compose.local.yml 핵심 설정
volumes:
  - .:/app              # 호스트 소스코드를 컨테이너에 마운트 (HMR 동작)
  - /app/node_modules   # node_modules는 컨테이너 내부 것 사용 (충돌 방지)
command: ["yarn", "run", "dev", "--host"]
environment:
  CHOKIDAR_USEPOLLING: "true"   # 파일 변경 감지 (macOS/Windows 파일시스템 이슈 대응)

```

### 실행 명령
```terminaloutput
docker compose -f docker-compose.local.yml up
# → localhost:5173 접속
```

### HMR 동작 원리:
호스트 파일을 컨테이너에 볼륨 마운트 → Vite가 변경 감지 → 브라우저 자동 갱신
