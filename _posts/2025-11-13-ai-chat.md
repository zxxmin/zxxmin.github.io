# 들어가며
사내 솔루션에 AI Assistant 기능을 작업하면서, AI 백엔드와 어떤 방식으로 통신할지 결정해야 했다.
ChatGPT처럼 응답이 실시간으로 타이핑되며 나오는 UX를 구현하려면 단순 HTTP 요청/응답 구조로는 한계가 있었다.

SSE, WebSocket, HTTP Polling 세 가지 방식을 비교 검토하고 최종적으로 SSE를 선택한 과정을 정리했다.

---

# 스트리밍 기능이 왜 필요했는가
우리가 작업하는 AI Assistant 기능은 우리 솔루션을 사용하는 회사 맞춤형이라 단순 FAQ 챗봇이 아니라, **해당 회사의 DB에 접근해 데이터를 조회하고 분석하는 형태**로 동작한다.

자연스럽게 ChatGPT와 유사한 UX가 요구됐다. 질문을 입력하면 AI가 실시간으로 답변을 생성하는 방식이다. 그런데 LLM은 응답을 한 번에 내려주는 게 아니라, **토큰 단위로 순차적으로 생성**한다.
이 토큰들을 어떻게 클라이언트까지 전달하느냐가 핵심 문제였다.

---

# 세 가지 통신 방식 검토
## HTTP Polling
클라이언트가 일정 주기로 서버에 요청을 반복하는 방식.

setInterval로 구현할 수 있지만, AI 응답이 완료됐는지 매번 확인해야 한다.
불필요한 요청이 쌓이고, 실시간으로 타이핑되는 느낌을 구현하기 어렵다.

## ✅ SSE (Server-Sent Events)
서버 → 클라이언트 단방향 스트리밍 방식.

HTTP 기반이며, 서버가 응답을 나눠서 순차적으로 전송할 수 있다.
LLM이 토큰을 생성하는 즉시 클라이언트에 밀어주는 구조에 적합하다.

## WebSocket
클라이언트 ↔ 서버 양방향 통신 방식.

실시간 동기화, 멀티유저 협업 등에 적합하지만, 연결을 지속적으로 유지해야 하므로 단순 스트리밍 응답에 쓰기에는 구조가 과하다.

---

# 세 가지 방식 비교
| 항목 | HTTP Polling | ✅ SSE                | WebSocket |
|------------------|--------------------|----------------------|----------------|
| 통신 방향 | 클라이언트 → 서버 (반복 요청) | 서버 → 클라이언트 (단방향)     | 양방향 |
| 프로토콜 | HTTP | HTTP                 | 별도 WS 프로토콜 (TCP) |
| 실시간 스트리밍 | 어려움 (폴링 주기에 의존) | 적합                   | 가능 |
| 서버 연결 유지 비용 | 낮음 | 낮음                   | 높음 |
| 구현 복잡도 | 낮음 | 보통                   | 높음 |
| AI 응답 스트리밍 적합도 | 낮음 | 높음                   | 가능하나 과한 구조 |
| 다중 클라이언트 동기화 | 불가 | 어려움                  | 적합 |
| HTTP 메서드 표준 지원 | GET/POST 모두 | GET (표준), POST (커스텀) | 해당 없음 |
| 브라우저 지원 | 모든 브라우저 | 광범위                  | 대부분 지원 |

# SSE를 선택한 이유
## GPT는 SSE와 WebSocket을 함께 사용
GPT는 두 방식을 **상황에 따라 혼합**해서 사용한다.

- **SSE**: 일반적인 텍스트 응답 스트리밍에 사용. LLM이 생성한 토큰을 순차적으로 화면에 출력하는 용도.
- **WebSocket**: 저지연 실시간 상호작용이 필요한 경우에 사용. 구체적으로는 두 가지 목적이 있다.
  - **실시간 음성/텍스트 상호작용 (Realtime API)**<br/>
    음성 입력과 텍스트를 실시간으로 주고받는 기능에서 WebSocket을 통해 저지연 양방향 통신을 구현했다.
  - **다중 클라이언트 동기화**<br/>
    같은 계정으로 여러 탭, 창, 디바이스에서 동일한 대화를 열었을 때, 한쪽에서 입력하면 다른 쪽도 즉시 업데이트된다. 여러 클라이언트가 하나의 대화 상태를 공유해야 할 때 WebSocket이 필요하다.

## 우리가 WebSocket을 선택하지 않은 이유
GPT가 WebSocket을 도입한 두 가지 조건이 우리 서비스에는 해당하지 않았다.

- **실시간 음성 기능이 없다.** 텍스트 입력과 텍스트 응답만 처리한다. 저지연 양방향 통신이 필요한 음성 상호작용이 없으므로 WebSocket의 핵심 이점이 적용되지 않는다.
- **다중 클라이언트 동기화가 필요 없다.** 한 사용자가 하나의 세션에서 질문하고 응답을 받는 단순 구조다. 여러 탭이나 디바이스 간 대화 동기화 요구사항이 없었다.
- **단방향 스트리밍으로 충분하다.** 클라이언트가 질문을 POST로 보내고, 서버가 응답을 스트리밍으로 내려주는 구조에서 양방향 통신은 필요하지 않다.
- **서버 연결 유지 비용을 고려했다.** WebSocket은 연결을 지속적으로 유지해야 하므로 동시 사용자가 늘어날수록 서버 부담이 커진다.

GPT가 WebSocket을 쓰는 이유는 명확하다. 하지만 그 이유가 우리 서비스에는 해당하지 않았다.
기술적으로 더 강력해 보이는 방식보다 **요구사항에 맞는 최소 복잡도의 방식**이 더 적합한 선택이었다.

결론적으로 **SSE + POST 방식**을 선택했다.

# 구현 과정
## SSE 표준의 한계와 해결
SSE 표준은 **GET 방식**을 기본으로 지원한다. 
하지만 AI 채팅 특성상 사용자의 질문을 Request Body에 담아야 했고, GET으로는 Body 전송이 어렵다. 
URL에 파라미터가 노출되는 보안 문제도 있었다.

이를 해결하기 위해 **POST 요청 + `text/event-stream` 응답** 조합을 사용했다. 
클라이언트는 일반 POST 요청을 보내고, 서버는 `text/event-stream` 형식으로 응답을 스트리밍한다.

## 전체 흐름
```
사용자 입력
    ↓
React → POST 요청 (JSON body, Accept: text/event-stream)
    ↓
FastAPI → LLM 호출 (스트리밍)
    ↓
FastAPI → SSE 형식으로 토큰 단위 응답 전송
    ↓
React → ReadableStream으로 수신 및 화면 렌더링
```

## 백엔드 응답 형식 (SSE)
SSE 표준 이벤트 형식을 그대로 따른다. 이벤트는 `\n\n`으로 구분되며, 각 이벤트는 `data:` 접두사를 가진다.
```
data: 안녕\n\n
data: 하세요\n\n
data: !\n\n
```

## 프론트엔드 구현 (React + TypeScript)
`fetch` API의 `ReadableStream`을 활용해 스트리밍 응답을 처리했다.
```javascript
export const useStreamHooks = async (
      url: string,
      body: StreamRequest,
      onMessage: (msg: string) => void,
      onDone?: () => void,
) => {
    const response = await fetch(`${BASE_URL}${url}`, {
        method: "POST",
        headers: {
            "Content-Type": "application/json",
            Accept: "text/event-stream",
        },
        body: JSON.stringify(body),
    });
    if (!response.body) {
        throw new Error("streaming response가 없습니다.");
    }
    const reader = response.body.getReader();
    const decoder = new TextDecoder("utf-8");
    let buffer = "";
    while (true) {
        const { value, done } = await reader.read();
        if (done) {
              onDone?.();
              break;
        }
        buffer += decoder.decode(value, { stream: true });
        // SSE 표준: 이벤트는 \n\n으로 구분
        const chunks = buffer.split("\n\n");
        // 마지막 chunk는 완전하지 않을 수 있으므로 버퍼에 보관
        buffer = chunks.pop() ?? "";
        for (const chunk of chunks) {
            if (!chunk.trim()) continue;
            
            // data: 라인 찾기
            const dataLine = chunk
                            .split("\n")
                            .find((line) => line.startsWith("data:"));
            if (!dataLine) continue;
            
            const payload = dataLine.replace(/^data:\s*/, "").trim();
            if (payload) {
                onMessage(payload);
            }
        }
    }
};
```

### 구현 시 주의한 포인트
- `Accept: text/event-stream` 헤더를 명시해야 서버가 스트리밍 응답으로 처리한다.
- `response.body.getReader()`로 스트림을 직접 읽는다.
- 네트워크 청크가 SSE 이벤트 경계와 일치하지 않을 수 있어 `buffer`로 누적 후 `\n\n` 기준으로 분리한다.
- `decoder.decode(value, { stream: true })` 옵션으로 멀티바이트 문자(한글 등)가 청크 경계에서 깨지지 않도록 처리한다.

# 결과
POST + SSE 방식으로 AI 어시스턴트 채팅 기능을 정상 구현했다. LLM이 토큰을 생성하는 즉시 클라이언트 화면에 순차적으로 출력되어 ChatGPT와 유사한 UX를 구현할 수 있었다.

프론트엔드는 기존 `fetch` API만으로 구현이 가능했고, 별도 라이브러리 의존성 없이 스트리밍 처리를 완성했다.

# 정리
- **WebSocket이 기술적으로 더 강력해 보여도, 요구사항에 맞는 최소 복잡도의 방식을 선택**하는 게 중요하다. GPT가 WebSocket을 쓰는 이유는 명확하지만, 음성 기능이나 다중 클라이언트 동기화가 없는 단순 AI 채팅이라면 SSE로 충분하다.
- SSE 표준은 GET을 기준으로 하지만, **POST + `text/event-stream` 조합이 실제 AI 채팅 구현에서는 더 현실적인 선택**이다. 표준 동작 방식과 실제 요구사항의 차이를 미리 파악해두는 것이 좋다.
- 스트리밍 응답을 파싱할 때 **청크 경계 처리(buffer 누적)**를 반드시 고려해야 한다. 한글처럼 멀티바이트 문자가 포함된 경우 청크 경계에서 깨짐이 발생할 수 있다.
